# “享租管家” SaaS 房产管理系统：业务功能与逻辑规格说明书

本档旨在将 `init_housing.sql` 中定义的数据库结构转化为清晰的业务功能矩阵与逻辑规则。这套系统以“多租户隔离、财务合规、物联自动化”为核心设计原则。

---

## 1. 机构与内控模块 (Org & Internal Control)
**核心逻辑：SaaS 物理隔离与精细化授权。**

- **多租户隔离**：所有业务单据强制绑定 `org_id`，确保不同机构（二房东/公寓商）数据物理不可见。
- **机构个性化配置**：通过 `orgs.sys_config` (JSON) 动态配置各家公司的业务规则（如：逾期几天算违约、催缴短信自动发送开关、滞纳金费率等）。
- **扁平化权限直派**：
    - 管理员可针对每个员工 (`org_staffs`) 勾选互不干扰的功能码 (`permission_code`)。
    - 无需定义角色等级，直接按需分配（如：店长 A 拥有财务审核权，而管家 B 只有录入权）。
- **全局操作审计**：
    - 系统级拦截器记录针对 `leases`, `bills`, `payments` 的任何增删改操作。
    - 留存修改前后的 `JSON` 快照，确保财务变更可追溯至具体责任人。

## 2. 房源资产管理 (Asset Management)
**核心逻辑：资产数字化与管家责任制。**

- **小区-房源二级结构**：支持多项目 (`projects`) 管理。
- **房源属性标签化**：
    - 预设全局配套设施库 (`sys_amenities`)，如家电配套、家具配套。
    - 通过关联表快速勾选房源配置，生成前端数字化详情页。
- **管家分配逻辑**：每间房可指派一名责任管家 (`manager_id`)。
    - 权限系统可据此过滤数据，实现“管家仅能管理自己名下的房源”。
- **房源状态生命周期**：自动记录 `空置 -> 预定(线下) -> 已租 -> 维修中 -> 下架` 的状态流转日志，生产房源空置率报表。

## 3. 租客与 C 端生态 (Tenant & CRM)
**核心逻辑：实名身份与全球通行账号解耦。**

- **B 端实名库**：记录合同签署人的真实姓名、身份证、手机号。
- **C 端登录账号**：租客通过微信登录产生的 `openid`/`unionid` 存放在 `tenant_users`。
- **动态绑定机制**：通过手机号或手动绑定，将“玩手机的人”与“签合同的人”关联。
    - **逻辑优势**：即使租客换了手机号登录，只要底层 ID 绑定，其名下在多个机构租的所有房子都能一站式查看。

## 4. 租约生命周期 (Lease Lifecycle)
**核心逻辑：版本链条追踪与财务清算闭环。**

- **租约版本链**：
    - 使用 `previous_lease_id` 串联。续租、换房不修改原合同，而是产生新租约并记录变动类型。
    - 确保历史账单与当时合同金额的绝对一致性。
- **同住人管理**：
    - 挂载多名同住人。支持为同住人绑定独立的 C 端账户，仅开放门禁开锁与报修权限。
- **多维度押金本**：
    - 将押金细分为：房屋押金、水电押金、钥匙押金等。
    - 记录：`应收 -> 实收 -> 下一次抵扣 -> 最终退还` 的完整生命周期。
- **退房清算逻辑 (`lease_checkouts`)**：
    - 离场时一键汇总：未付账单 + 违约扣款 + 物品损耗。
    - 与押金本自动对冲，生成 `最终应退/应补` 金额，经财务审核后执行资金下发。

## 5. 财务与对账引擎 (Financial Engine)
**核心逻辑：权责发生制记账与资金真实流向分离。**

- **账单生成逻辑**：
    - 房租账单：根据 `payment_cycle` (如付三) 自动或手动预生成。
    - 水电账单：关联抄表记录或充值记录生成。
- **多通道收款配置**：
    - 支持机构配置各自的微信特约商户号（线上 API）或企业银行账户（线下核销）。
    - 租约级别可锁定支付通道（如：指定某笔租约只能线下转账）。
- **资金流水对账 (`payment_records`)**：
    - 记录真实到账时间、手续费扣除、净入账。
    - 支持一笔资金核销多个账单。
- **全成本核算**：
    - **底租成本**：计算给原房东的应付账单。
    - **中介佣金**：记录每单交易的获客成本。
    - **外部支出**：记录给开发商、物业的各种管理开支。
    - **利润计算**：系统汇总得出“单房/单项目/单机构”的真实净利润。
- **内部资金调拨**：跟踪资金从微信商户账户提现到公司对公户的流向。

## 6. 物联与智能化服务 (IoT & Utilities)
**核心逻辑：抄表防纠纷与预付费两阶段闭环。**

- **抄表与取证**：记录所有表计的指针读数，并支持上传现场照片防赖账。
- **两阶段充值下发**：
    - 针对预付费电表：租客交钱（阶段一）-> 系统向 IoT 网关下发指令（阶段二）。
    - 专门使用 `iot_sync_status` 追踪指令是否到达硬件，失败则触发报警。

## 7. 租约生命周期操作详解 (Lease Operations Deep Dive)

本章详细说明 SaaS 后台执行核心业务操作时的底层逻辑步序。

### 7.1 新签租约 (New Lease)
**场景**：外部新客入驻或空置房源重新出租。
1.  **房源校验**：检查 `rooms.status` 是否为 `0(空置)` 或 `2(预定)`。
2.  **创建租客**：若为新客，在 `tenants` 建立实名档案；若已存在，则复用并建立与 `tenant_users` 的绑定。
3.  **生成租约**：在 `leases` 插入主记录，初始化 `status=1(生效中)`，`change_type=0(新签)`。
4.  **初始化账务**：
    - 根据 `payment_cycle` 立即预生成首期 `bills` (租金)。
    - 根据合同要求生成 `bills` (押金账单)。
    - 在 `lease_deposits` 初始化押金账本。
5.  **锁定房源**：同步更新 `rooms.status=1(已租)`，并向 `room_status_logs` 写入变更证据。

### 7.2 续租 (Renewal)
**场景**：租客合同即将到期，签署补充协议延长租期或调整价格。
1.  **前置状态**：原租约 `status` 必须为 `1(生效中)`。
2.  **创建新本**：在 `leases` 插入新行。
    - `previous_lease_id` 指向原租约 ID，形成版本链。
    - `change_type` 标记为 `1(续租)`。
3.  **资金结转**：
    - 检查原租约 `lease_deposits` 余额，若租金上涨，则生成“补缴押金”的补差账单。
    - 标记原合同在到期时自动转为 `2(已到期)`。
4.  **下发通知**：通过 `sys_message_logs` 向租客小程序发送续租签署成功的通知。

### 7.3 换房结算 (Room Change / Transfer)
**场景**：租客从 A 房搬至 B 房（同机构内）。
1.  **目标房锁定**：检查目标房 `rooms.status=0`。
2.  **原房释放**：
    - 记录原房 `meter_readings` (最终读数)。
    - 将原房 `rooms.status` 置为 `3(维修/保洁中)`。
3.  **创建新租约**：
    - `previous_lease_id` 指向旧租约，`change_type` 标记为 `2(换房)`。
4.  **财务对冲**：
    - **逻辑关键**：将旧租约未使用的租金余额转为新租约的首期抵扣项。
    - 押金在 `lease_deposits` 中执行“跨租约转移”。
5.  **权限转移**：自动注销原房智能锁密码，同步向新房下发新密码。

### 7.4 正常退房结算 (Normal Checkout)
**场景**：合同到期，租客搬走，结清余款。
1.  **取证录入**：管家录入退房当天的 `meter_readings` (水电指针) 和 `lease_documents` (退房照片)。
2.  **生成清算单 (`lease_checkouts`)**：
    - **退还项**：读取 `lease_deposits.amount_paid` (实收押金)。
    - **扣款项**：计算末期水电费 + 历史欠费 + 物品损耗补偿。
    - **差额计算**：`最终退还 = 总押金 - 总扣费`。
3.  **审核流**：进入 `review_status=0(待核准)`，由财务确认通过后，执行原路退款。
4.  **房源释放**：`rooms.status` 回归 0 或 3。

### 7.5 违约/提前退租 (Breach Checkout)
**场景**：租客中途违约或机构根据条款强制清退。
1.  **触发惩罚逻辑**：
    - `change_type` 标记为 `4(违约退租)`。
    - 自动根据 `orgs.sys_config` 中的违约模板（如：扣除所有押金或额外罚 1 个月租金）计算 `penalty_amount`。
2.  **欠费强扣**：在 `lease_checkouts` 中，优先使用押金本余额 (`amount_paid`) 抵扣所有未付的 `bills`。
3.  **文档存证**：强制上传违约通知书或现场取证照片至 `lease_documents`。
4.  **黑名单联动**：(可选逻辑) 标记该租客实名档案为 `status=0(异常)`，限制其在平台内其他机构租房。

### 7.6 租约动态调整 (Lease Adjustment)
**场景**：租约进行中，临时决定降租、加人、或改变计费模式。
1.  **记录版本变动**：在 `leases` 插入调整记录（ChangeType=3），确保不覆盖原合同快照。
2.  **账单重置**：
    - 作废原租期内“尚未支付”的未来账单。
    - 按照新价格重新生成剩余租期的账单模板。
3.  **配置生效**：如调整了 `billing_type`（先付改后付），后续生成的电费对账单逻辑将跟随变动。

### 7.7 账务账单专项操作 (Financial Billing Operations)
**场景**：处理大额账单分期、账单拆分以及每月的自动化出账任务。

#### 7.7.1 账单拆分与分期 (Splitting & Installments)
1.  **锁定原单**：选择一笔 `status=0(未支付)` 的原始账单作为 `parent_bill_id`。
2.  **创建子单**：
    - 根据分期计划，创建多笔新 `bills` 记录。
    - 将 `parent_bill_id` 设为原账单 ID，`batch_no` 设置为同一分期批次号。
3.  **原单作废/核销**：
    - 获取原账单金额，确保所有子单总和等于原单。
    - 将原账单 `status` 置为 `3(作废)`，并在备注中记录“已拆分为子单 ID: [xxx, yyy]”。

#### 7.7.2 自动化月度扣费任务 (Batch Billing Generation)
1.  **扫描引擎**：系统任务定时扫描所有 `leases.status=1(生效中)` 且 `billing_type=2(后付)` 或固定租金类租约。
2.  **出账判断**：
    - 计算 `next_billing_date`（根据 `start_date` 和 `payment_cycle` 推算）。
    - 检查是否已存在同周期的 `bills`。
3.  **批量创建**：
    - 自动插入批量账单，设置 `review_status=0(待审核)`（根据机构配置决定是否免审）。
4.  **触达通知**：生成对应的 `sys_message_logs` 记录，通过微信/短信推送“本月租金待缴纳”提醒。

---

> [!TIP]
> **设计哲学小结**：
> 1. **数据不可逆**：财务和合同的关键变更全部通过“增量日志/新版本”实现，严禁直接 Update 核心数值。
> 2. **职责明确**：钱归钱（Records）、账归账（Bills）、证据归证据（Readings/Documents）。
> 3. **极简权限**：跳过角色控制，直接面向员工职能配置，适配各级公寓机构。
